# 仪式 · 接入上游文档

人产文档（PRD / 设计稿 / 技术设计）怎么进机制。原则一句话：**项目文档树只保存"锚 + 摘编 + 账本"，永不复制全文。**

## 先分级

| 级别 | 例子 | 待遇 |
|---|---|---|
| 权威源 | 定稿 PRD、评审过的设计稿、拍板的技术设计 | 走下面三步 |
| 原始素材 | 纪要、聊天记录、口头需求 | 落 `now/inbox/` 等 CH 裁决（转 `rituals/change.md`），永不直接作为实现依据 |

## 权威源三步

1. **挂锚**：`truth/sources/index.md` 记一行（模板 templates/sources-index.md）：编号 S-##（全局递增）| 名称 | 锚 | 版本 | 定稿日期 | 负责人。
2. **摘编 digest**（模板 templates/digest.md，≤150 行）：关键需求 / 约束 / 验收标准，每条回链原文章节号。AI 起草，**人确认后才算数**——它是对齐契约：管对齐，不管完整。
3. **条目账本 items**（模板 templates/items.md）：把源拆成可验收条目，每条一行，四状态：未排期 / 进行中(T-###) / 已实现(证据) / 明确不做(D-### 或 CH-###)。**渐进细化**：0→1 大 PRD 先拆章节级粗条目，哪块临近开工再细化那一章——任何时点对源的覆盖都是完整的。

## 各类型路径

- **PRD**：外部工具维护 → 挂锚 + 摘编 + 账本。
- **设计稿**（Figma 等）：挂锚 + 摘编（交互规则 / 状态清单 + 关键帧图链）。
- **技术设计**：推荐直接写 `truth/modules/<模块>`（省一层摘编，评审走 PR diff）；团队习惯在外部写时挂锚 + 摘编兜底。
- **评审会**：只留结论进 decisions.md 或模块真相；会议过程永不进真相层。

## 读的纪律

- 实现时不读摘编读原文：任务包按 covers 条目把**原文章节完整细节**拉进任务包（只拉自己那一片）。
- 已知限制：原文锚点会随源改版漂移——经 CH 升版源时，人工核对波及条目的锚点。
