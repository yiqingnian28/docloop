# 仪式 · 记变更（inbox → CH → 裁决）

需求一变就走这条流水线。**从"人说的话"到权威层只有这一条路。**

## 流水线

1. **收料**：纪要 / 聊天 / 口头需求原样落 `now/inbox/`（不加工；不进上下文，只是 CH 的附件线索）。
2. **起草 CH**（模板 templates/change.md，≤80 行）：提炼变化点 + **冲突检查报告**——对照条目账本 + truth（含 decisions）+ 本迭代 CH，逐条列出"与 S-##.R-### 矛盾 / 与 CH-### 裁决相反"；无冲突也要写"未发现"。跨迭代旧 CH 不是检查面（长期裁决在结算时已沉淀进 decisions 与账本状态）。
3. **人裁决**：PM / 负责人拍板哪个算数；CH 记录裁决人姓名与结论。**未裁决前不许改权威层。**
4. **改写权威层**（按裁决结论，顺序）：源升版本（登记簿）→ 摘编 / 账本改写 → 波及 truth 改写 → 任务包调整。truth 被改写成新真相，自己不记流水（diff 在 git）。

编号规则：CH-### / T-### 全局递增永不复位——新号 = 全仓（含 past/）最大号 + 1。

## 处置分两种

- **本迭代改**：调 iteration.md 任务清单 + 受影响任务包记偏差。
- **挂起待排**：账本条目置"未排期"并回链 CH（只写 ID），体检盯着不让丢。

## 反向（实现中发现设计要改）

走 sdd-riper Reverse Sync 直接回改 truth，任务包记偏差；**当设计变化源于需求理解变化时，补一张 CH**。

## 三条底线

inbox 不进上下文；裁决有名有姓（翻案考古有据）；未裁决 ≠ 不存在（inbox 积压体检标黄）。
