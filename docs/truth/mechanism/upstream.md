# 上游接入：人产文档怎么进机制

> 上级：[总述](README.md)。现实场景：PM 出产品文档、设计师出设计稿、技术出技术设计，加评审环节，人参与度高。这些文档在别人的工具里活着（飞书 / Word / Figma），体积不受控、版本会变、还可能互相矛盾。
> 设计原则一句话：**上游文档是外部真相源，项目文档树只保存"锚 + 摘编 + 账本"，永不复制全文。**

## 信源分级：不是所有人产文档都平等

| 级别 | 例子 | 待遇 |
|---|---|---|
| **权威源** | 定稿 PRD、评审通过的设计稿、拍板的技术设计 | 进登记簿、有版本号、产摘编 + 账本，是 AI 的实现依据 |
| **原始素材** | 会议纪要、聊天记录、口头需求、临时想法 | 落 `now/inbox/`，**永不直接作为实现依据**，等待 CH 裁决（见 [change.md](change.md)） |

## 挂锚：登记簿

上游文档产出 / 定稿 → `truth/sources/index.md` 记一行：

```
编号 | 名称 | 锚（链接/路径） | 版本 | 定稿日期 | 负责人
S-03 | 销售模块PRD | <飞书链接>    | v2.4 | 2026-07-01 | 张PM
```

已知限制：原文锚点（如 §4.2）会随源改版漂移，lint 查不了外部锚——**经 CH 升版源时，人工核对波及条目的锚点**。

## 摘编（digest.md）：管"对齐"，不管"完整"

- 每个需要 AI 消化的源，产一份 `sources/S-##_<名>/digest.md`：关键需求、约束、**验收标准**，每条带回链原文章节号。体积受限（默认 ≤150 行），改写不追加。
- 摘编由 **AI 起草、人在评审环节确认**——它是人和 AI 之间的**对齐契约**：人确认"这 150 行就是 PRD 的意思"，AI 后续按摘编交付才有握手依据。
- 100 页 PRD 在 AI 必读集里永远只占 150 行。

## 条目账本（items.md）：管"完整"，细节不许默默消失

摘编是有损压缩，装不下大 PRD 的全部细节——完整性不靠它，靠账本**枚举对账**：

```
S-03.R-042 | 订单支持部分发货 | →原文§4.2 | 进行中(T-012)
```

- 把源拆成一条条**可验收的需求条目**，每条一行。它是清单不是散文，天然抗臃肿（100 页 PRD 拆 300 条也就几百行）。
- 每条四种状态：**未排期 / 进行中(T-###) / 已实现(验收证据链接) / 明确不做(决策链接)**。
- truth 层（含本账本）引用任务 / 变更单**只写 ID 不写路径**（T-###、CH-###）——now/ 结算后搬进 past/，路径链接必断；lint 按 ID 在 now/ 与 past/ 两处解析。
- **渐进细化**：0→1 大 PRD 不必一次拆完——挂锚时先拆到章节级粗条目，哪个模块临近开工再细化那一章。任何时点账本对总量的覆盖是完整的（粗粒度也是覆盖）。
- 账本过大 → 按章节拆目录（同模块分层规则）。

## 完整性三道闸

1. **实现时不读摘编读原文**：开任务包时，把该任务覆盖条目对应的**原文章节完整细节**拉进任务包（只拉自己那一片）。摘编是导航，**从来不是实现依据**。
2. **结算时对账**：迭代结算硬检查——本迭代范围内的条目，每条要么有任务 + 证据，要么显式挂起 / 否决。没有一条允许默默消失。
3. **体检盯孤儿**：lint 扫描长期无归属的条目、任务声称完成但条目没勾的（见 [budget-lint.md](budget-lint.md)）。

## 三类人产文档各自的路径

| 类型 | 路径 |
|---|---|
| **产品文档（PRD）** | 外部工具维护 → 挂锚 + 摘编 + 条目账本 |
| **设计稿（Figma 等）** | 没法进仓库 → 挂锚 + 摘编（摘编里写交互规则 / 状态清单，关键帧导出图片链接） |
| **技术设计文档** | 推荐**直接以 `truth/modules/<模块>` 为载体**在仓库写（省一层摘编，评审走 PR diff）；团队习惯在外部写时走挂锚 + 摘编兜底 |
| **评审环节** | 只留结论：决策进 `decisions.md` 或模块真相；需留痕的一页纪要落 now/ 随迭代冻结。**会议过程永不进真相层** |

## AI 阅读协议（渐进披露）

默认只读摘编；需要细节时按锚点**深读原文的特定章节**；禁止把上游全文当常驻上下文。摘编计入开工必读集预算。
