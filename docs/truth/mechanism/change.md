# 变更管理：CH 单 + inbox + 裁决流水线

> 上级：[总述](README.md)。这是机制对"需求频繁变更 → 文档臃肿"痛点的正面回答：**变更是一等公民，有自己的家；真相文档只被改写，不记流水。**

## 变更单（CH）

一次需求变化 = 一张 `now/changes/CH-###_<名>.md`（默认 ≤80 行），只记三段：

| 段 | 内容 |
|---|---|
| **变化** | 需求怎么变的（3–5 句）+ 上游出处链接（源 ID / inbox 素材） |
| **波及** | truth/ 哪几份要改（链接）· 哪些条目 / 任务包受影响（链接） |
| **处置** | 本迭代改 or 挂起待排 · 裁决人 + 结论 · 完成勾选 |

传导链集中在这一张纸上；truth 被直接**改写成新真相**（diff 在 git 里），自己不记账。迭代结算时 CH 随 now/ 冻结进 past/，summary 里列一行清单。**变化有自己的家，真相文档永远瘦。**

## 裁决流水线：从"人说的话"到"权威层"只有一条路

人是不可靠的：纪要会互相矛盾、会出错。机制不指望人变可靠，而是**把矛盾显式摆上桌面、逼着裁决完才能入库**：

```
纪要/聊天说"要改X"
  → 落 now/inbox/（原样存·不加工）
  → AI 起草 CH 单：提炼变化点 + ★冲突检查报告★
      （对照现有条目账本 + truth + 历史 CH 裁决，明确列出：
       "这条和 S-03.R-042 矛盾"、"和上周 CH-005 的裁决相反"）
  → 人裁决（PM/负责人拍板哪个算数）
  → 裁决结果才改写权威层：
      源升版本（登记簿）+ 摘编/账本改写 + 波及的 truth 改写 + 任务包调整
  → CH 单记录裁决人和结论，随迭代冻结
```

**人机分工**放在各自擅长的位置：AI 擅长大海捞针找矛盾（新纪要 vs 几百条账本逐条比对，人做必漏）；人擅长拍板（两个说法冲突时哪个算数，只有人有权定）。

## 三条配套规矩

1. **inbox 不进上下文**：纪要原文只是 CH 单的附件线索，AI 开工必读集里没有它，随迭代冻结。
2. **裁决有名有姓**：CH 单必须记"谁拍的板"，将来翻案时考古有据。
3. **未裁决 ≠ 不存在**：inbox 里躺着没处理的素材，体检标黄提醒——防"开会说了但没人落单"的经典漏项。

## 与实现的接口

- 裁决"本迭代改" → 调整 `iteration.md` 任务清单 + 受影响任务包记偏差；
- 裁决"挂起待排" → 条目账本置"未排期"并回链 CH，体检盯着；
- 实现中发现设计要改（反向）→ 走 sdd-riper Reverse Sync 回改 truth，必要时补一张 CH（当设计变化源于需求理解变化时）。
