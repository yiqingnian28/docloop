# lint 需要识别当前规则被证据和裁决噪声淹没的问题

## 背景

在把复杂设计文档拆成“主阅读面 / 细节工具书面 / 追溯面”时，新增的“历史痕迹密度”检查可以发现显式的修订历史、时间线、旧口径堆积。但实际迁移中还出现了另一类问题：正文没有明显的“修订历史”标题，却用长段落把当前规则、裁决编号、日期、机器预检证据、代码锚点、排期估算混在一起。

结果是读者仍需要先问“这段存在的意义是什么”，再人工把它改成“当前规则表”。这说明机制已经抓住了大方向，但还没有覆盖“当前规则被证据/裁决噪声淹没”的阅读面问题。

## 为什么现机制没发现

- 项目尚未正式落 `docs/truth/now/past` 时，只能做等价映射；当前 lint 不方便扫描这类 `truth-like` 路径。
- 第 7 项主要识别历史标题和历史词密度，但很多噪声不是“历史词”，而是证据、代码锚点、裁决 ID、日期、排期和实现状态混排。
- 机制没有显式定义 truth / 细节工具书正文的 section contract：哪些句子是当前规则，哪些只应该是追溯链接，哪些证据应去 task / changelog / past。
- “编码现状 / 机器预检 / 证据明细 / 附C / 文件:行号 / 人天估算”等内容可以合法停留在正文里，导致工具书面仍然像审计报告。

## 建议改进

1. 支持配置 `truthLikePaths` 或 `scanPaths`，让未完全迁移到 `docs/truth` 的项目也能被第 7 项和后续阅读面 lint 扫到。
2. 增加“阅读面噪声”检查：
   - 单条 bullet / 表格单元过长；
   - 一行内出现多个 ID、日期、括号解释或代码位置；
   - truth 正文出现“机器预检 / 证据 / 附C / 编码现状 / 人天 / 待落码”等过程性词；
   - 同一段同时包含当前规则、证据、排期或追溯信息。
3. 给出可执行 rewrite recipe：
   - 当前规则改成 `规则 / 当前口径` 表；
   - 代码锚点与机器预检证据移动到 task / 开发任务 / 附录；
   - 裁决经过移动到 CHANGELOG / CH / past；
   - truth 只保留必要追溯 ID 或链接。
4. lint 输出不要只报“过程词密度高”，最好能提示“这段像证据明细，建议改成当前规则表 + 证据链接”。

## 验收想象

给一段混合了当前规则、裁决编号、机器预检证据、代码行号和排期估算的正文，docloop 能标黄并建议拆成：

- 当前规则表；
- 实现边界表；
- 追溯/证据链接。

这样可以减少人工追问“这段到底存在的意义是什么”的次数。
