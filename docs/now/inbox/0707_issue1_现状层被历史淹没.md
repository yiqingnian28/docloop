# 文档现状层被变更历史淹没,影响阅读与维护

## 问题现象

在长期迭代的文档中,当前设计说明、历史变更记录、裁决过程、待确认项、代码核验结果和修订说明容易混在同一份正文里。

这会导致几个问题:

- 人阅读时很难快速抓住"现在到底是什么口径"。
- 正文里充满"旧口径、已裁决、已作废、待确认、历史来源、某次会议变更"等过程信息,现状反而被淹没。
- 细节确实没有丢,但细节和历史一起堆在正文里后,文档不像当前说明,也不像资料库,阅读路径不清晰。
- 变更追溯的初衷是好的,但如果正文承担过多追溯职责,会让每次阅读都变成考古。
- 文档越维护越长,后续 AI 或人类协作者都更难判断哪部分是当前事实、哪部分只是历史痕迹。

## 可能原因

- 当前事实、变更历史、评审裁决、待办状态没有严格分层。
- 为了保证可追溯,每次变更倾向于在正文里追加解释,而不是改写当前口径并把历史放到独立追溯面。
- 同一条事实可能同时存在于现状文档、变更记录、会议纪要、任务看板和细节说明中,导致维护成本上升。
- 文档缺少体积预算或历史痕迹密度检查,导致"可追溯"逐渐演变成"正文历史化"。
- 细节文档缺少资料库式组织方式,导致细节、原因、历史和当前结论混排。

## 改进建议

- 强化"当前口径"和"历史追溯"的边界:正文优先呈现当前事实,历史通过独立机制追溯。
- 给现状文档设置更明确的阅读目标:让读者能先快速理解当前设计,再按需查细节。
- 细节内容可以保留,但应更像工具书或资料库:便于查阅,而不是把所有演进过程都放在主阅读路径上。
- 变更历史建议通过专门的追溯面承载,正文中只保留必要引用。
- 增加文档体检规则,识别历史痕迹过密、正文过长、当前事实被过程信息淹没等问题。
- 明确每类信息的唯一权威位置,减少同一事实多处认真维护。
